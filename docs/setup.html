Local Setup

Get Granite-IAM running on your local machine in under 5 minutes. This guide covers prerequisites, infrastructure setup, building the project, and verifying everything works.

Prerequisites

Before you begin, make sure you have the following tools installed:

Tool Version Purpose Install
Java (JDK) 21+ Runtime environment Adoptium
Docker 24+ Container runtime Docker Desktop
Docker Compose v2+ Service orchestration Included with Docker Desktop
Git 2.x Version control git-scm.com

💡 Tip: You do not need Gradle installed globally. The project includes the Gradle Wrapper (./gradlew) which downloads the correct version automatically.

Verify your installations:

java --version        # Should show 21.x
docker --version      # Should show 24.x or higher
docker compose version # Should show v2.x
git --version         # Should show 2.x

Clone & Configure

  1. Clone the repository 
    git clone https://github.com/Lelisayohanes/Granite-IAM.git
cd Granite-IAM
  2. Create your environment file

    Copy the example environment file and edit it with your preferred values:

    cp .env.example .env
  3. Edit .env with secure credentials 
    # .env
POSTGRES_DB=granite_iam
POSTGRES_USER=granite
POSTGRES_PASSWORD=your_secure_password_here   # ← change this!

    🚨 Important: Never commit your .env file to version control. It is listed in .gitignore by default. Use strong, unique passwords.

Start Infrastructure

The docker-compose.yml file defines the infrastructure services required by Granite-IAM. Start them in detached mode: 

# Start all services in the background
docker compose up -d

This starts the following services:

Service Image Port Health Check
PostgreSQL postgres:16 5432 pg_isready (auto)
Redis redis:7 6379 —

Verify the services are healthy:

docker compose ps

# Expected output:
# NAME                  IMAGE         STATUS       PORTS
# granite-iam-postgres  postgres:16   Up (healthy) 0.0.0.0:5432->5432/tcp
# ...                   redis:7       Up           0.0.0.0:6379->6379/tcp

✔ Note: PostgreSQL data is persisted in a Docker volume (postgres_data). Your data survives container restarts. To reset, run docker compose down -v.

Build & Run

  1. Build the project 
    # Build all modules (skip tests for faster startup)
./gradlew build -x test

# Or build with tests
./gradlew build
  2. Run the application 
    # Start the web module (Spring Boot application)
./gradlew :granite-iam-web:bootRun

    The application will start on http://localhost:8080 by default. Spring Boot DevTools enables automatic restart on code changes.

  3. (Optional) Run with specific environment variables 
    # Override defaults at runtime
POSTGRES_DB=my_custom_db ./gradlew :granite-iam-web:bootRun

💡 Hot Reload: The spring-boot-devtools dependency enables automatic restart when files on the classpath change. Just save your code and the app restarts within seconds.

Verify Setup

Once the application is running, verify everything is working:

  1. Check the application health 
    curl http://localhost:8080/actuator/health

# Expected response:
# {"status": "UP"}
  2. Verify database connection

    If the health endpoint returns "status": "UP", the application successfully connected to PostgreSQL. JPA will auto-create tables based on the ddl-auto: update setting.

  3. Access the OpenAPI documentation 
    # Swagger UI (interactive API explorer)
open http://localhost:8080/swagger-ui.html

# OpenAPI JSON spec
curl http://localhost:8080/v3/api-docs
  4. Check Prometheus metrics 
    curl http://localhost:8080/actuator/prometheus | head -20

⚠️ Troubleshooting: If the application fails to start, check that:

  • PostgreSQL and Redis containers are running (docker compose ps)
  • Port 5432 and 6379 are not occupied by other services
  • Your .env credentials match the application.yaml configuration
  • You're using Java 21 (java --version)

Stopping Services

# Stop the Spring Boot application (Ctrl+C in the terminal)

# Stop infrastructure services (data preserved)
docker compose down

# Stop and remove all data (full reset)
docker compose down -v

Deploying This Documentation to GitHub Pages

This documentation site is designed to be deployed via GitHub Pages with zero build steps. Follow these steps to publish:

  1. Push the /docs folder to your main branch 
    git add docs/
git commit -m "docs: add documentation site"
git push origin main
  2. Enable GitHub Pages in your repository settings
    • Go to your repository on GitHub
    • Navigate to Settings → Pages
    • Under Source, select:
    • Branch: main
    • Folder: /docs
    • Click Save
  3. Access your documentation

    After 1-2 minutes, your site will be live at:

    https://<your-username>.github.io/<repository-name>/

    For example: https://lelisay.github.io/Granite-IAM/

✔ Auto-deploy: Every push to the main branch that modifies files in /docs will automatically trigger a new deployment. No build pipeline or CI configuration needed.

💡 Custom Domain: To use a custom domain, add a CNAME file in the /docs folder containing your domain (e.g., docs.granite-iam.dev), then configure your DNS provider to point to GitHub Pages.
