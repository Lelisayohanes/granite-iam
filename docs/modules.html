Modules

Granite-IAM is organized as a Gradle multi-module project. Each module represents a distinct architectural layer with specific responsibilities, dependencies, and testing strategies.

Module Layer Dependencies Status
granite-iam-core Domain None (pure Java) Active
granite-iam-application Application core, Jakarta Validation In Progress
granite-iam-infrastructure Infrastructure core, Spring Boot BOM In Progress
granite-iam-web Web / API core, infrastructure, Spring Boot Active

granite-iam-core Pure Domain

🌱 The Heart of the System

The core module is the innermost layer of the architecture. It contains pure Java domain entities, value objects, and business rules with zero framework dependencies. This ensures the domain model remains portable, testable with plain JUnit, and free from infrastructure concerns.

Responsibilities

  • Define domain entities (User, and future entities like Role, Permission)
  • Encapsulate business rules and validation logic within entities
  • Provide immutable identity (UUID) and timestamp management
  • Define domain events for lifecycle changes

Current Entities

// com.lelisay.graniteiam.core.domain.User
public class User {
    private final UUID id;
    private String username;
    private String passwordHash;
    private String firstName;
    private String lastName;
    private boolean active;
    private final Instant createdAt;
    private Instant updatedAt;
}

Design Principles

Immutability Where Possible

The id and createdAt fields are final. Mutations go through well-defined setter methods that update the updatedAt timestamp.

No Framework Leakage

No JPA annotations, no Spring beans, no Lombok. The domain model is pure Java, making it trivially testable and framework-agnostic.

Reconstruction Pattern

The static reconstruct() method supports hydrating entities from persistence without triggering creation-side effects.

granite-iam-application In Progress

⚙ Use Case Orchestration

The application module contains use cases (application services) that coordinate between the domain layer and external concerns. It defines port interfaces that infrastructure adapters implement, following the Hexagonal Architecture pattern.

Responsibilities

  • Orchestrate multi-step operations (e.g., create user + publish event + send email)
  • Define input/output DTOs and command objects
  • Declare port interfaces for infrastructure dependencies
  • Handle transaction boundaries at the use-case level

Dependencies

// build.gradle
dependencies {
    implementation project(':granite-iam-core')
    implementation 'jakarta.validation:jakarta.validation-api:3.0.2'
}

Note: Jakarta Validation is used for command object validation without pulling in the full Spring framework. This keeps the application layer lightweight.

granite-iam-infrastructure In Progress

🖨 External Concerns Adapter

The infrastructure module implements the port interfaces defined in the application layer. It handles all external integrations: database persistence (JPA/PostgreSQL), caching (Redis), messaging (Kafka), and any third-party service calls.

Responsibilities

  • Implement repository port interfaces using Spring Data JPA
  • Map between domain entities and JPA entity objects
  • Manage Redis cache operations for sessions and tokens
  • Publish domain events to Kafka topics
  • Handle external API integrations (email, SMS, etc.)

Dependencies

// build.gradle
plugins {
    id 'java'
    id 'io.spring.dependency-management'
}

dependencyManagement {
    imports {
        mavenBom "org.springframework.boot:spring-boot-dependencies:3.2.0"
    }
}

dependencies {
    testImplementation 'org.springframework.boot:spring-boot-starter-test'
}

The module uses the Spring Boot BOM for dependency version management but doesn't directly apply the Spring Boot plugin — it's a library consumed by the web module.

granite-iam-web Composition Root

🌐 HTTP Layer & Application Entry Point

The web module is the outermost layer and the Spring Boot application entry point. It handles HTTP concerns: REST controllers, request/response mapping, security configuration, OpenAPI documentation, and application lifecycle management.

Responsibilities

  • Expose REST API endpoints via @RestController classes
  • Configure Spring Security (OAuth2 resource server, JWT validation, CORS)
  • Map HTTP requests to application use cases
  • Generate OpenAPI documentation via SpringDoc
  • Expose Actuator health checks and Prometheus metrics
  • Wire all modules together as the Spring Boot composition root

Key Dependencies

Dependency Purpose
spring-boot-starter-web Embedded Tomcat, MVC framework, JSON serialization
spring-boot-starter-security Authentication & authorization framework
spring-boot-starter-data-jpa JPA/Hibernate ORM for database access
spring-boot-starter-data-redis-reactive Reactive Redis client for caching
spring-kafka Apache Kafka integration for event streaming
spring-boot-starter-oauth2-resource-server JWT token validation for OAuth2
spring-boot-starter-validation Bean validation with Hibernate Validator
spring-boot-starter-actuator Health checks, metrics, and management endpoints
springdoc-openapi (2.3.0) Auto-generated Swagger/OpenAPI documentation
micrometer-registry-prometheus Prometheus metrics export
postgresql (runtime) JDBC driver for PostgreSQL
spring-boot-devtools Hot reload during development
