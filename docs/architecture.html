Architecture

Granite-IAM follows a clean layered architecture inspired by Domain-Driven Design (DDD), enforcing strict dependency rules between modules to maintain testability and long-term maintainability.

System Overview

Granite-IAM is an Identity and Access Management system designed around the principle of separation of concerns. Each Gradle module maps to a distinct architectural layer with well-defined responsibilities and dependency boundaries.

💡 Dependency Rule: Dependencies always point inward. The core module has zero framework dependencies. The web module is the composition root that wires everything together.

Layered Architecture

Web Layer — granite-iam-web REST Controllers Spring Security OpenAPI / Swagger Actuator Request Validation
↓ ↑
Application Layer — granite-iam-application Use Cases Service Orchestration DTOs Port Interfaces
↓ ↑
Infrastructure Layer — granite-iam-infrastructure JPA Repositories Redis Cache Kafka Producers External APIs
↓ ↑
Core / Domain Layer — granite-iam-core Domain Entities Value Objects Domain Events Business Rules

Dependency Direction

    ├── depends on → granite-iam-infrastructure

granite-iam-application
    └── depends on → granite-iam-core

granite-iam-infrastructure
    └── depends on → granite-iam-core

granite-iam-core
    └── (no dependencies — pure Java)

Data Flow

A typical request flows through the system from the HTTP layer down to persistence and back. Here's how a user registration request is processed:

  1. HTTP Request arrives at the Web layer. The controller validates the request payload using @Valid annotations and maps it to a DTO.
  2. Application Service receives the DTO, orchestrates business logic, and delegates to the Core domain model for entity creation and rule enforcement.
  3. Domain Entity (User) is created in the Core layer with UUID-based identity, validated password hash, and immutable timestamps.
  4. Infrastructure Adapter persists the entity via a JPA repository (PostgreSQL) and optionally publishes a domain event to Kafka.
  5. Response flows back up the stack. The result is mapped to a response DTO and returned as JSON with appropriate HTTP status codes.

Security Model

Granite-IAM uses Spring Security 6 configured as an OAuth2 Resource Server. The security model is designed for token-based authentication with JWT.

🔑 JWT Token Validation

Incoming requests are validated against a configured JWT issuer. Tokens must be signed and unexpired. Claims are mapped to Spring Security authorities.

👥 Role-Based Access Control

Endpoints are secured with role annotations (@PreAuthorize). Roles are extracted from JWT claims and enforced at the method level.

🔒 Password Hashing

Passwords are never stored in plain text. The User domain entity stores only the passwordHash, ensuring security at the model level.

🛡 CORS & CSRF

Stateless JWT authentication with CSRF disabled for API endpoints. CORS is configured to allow specific trusted origins in production.

Infrastructure

The system relies on three primary infrastructure services, all managed via docker-compose.yml for local development.

Service Role Configuration
PostgreSQL Primary relational database for user accounts, roles, and audit records Configured via application.yaml + env vars from .env
Redis In-memory cache for sessions, token blacklisting, and rate limiting Reactive client via spring-boot-starter-data-redis-reactive
Apache Kafka Event streaming for audit logs, async notifications, and domain events Spring Kafka auto-configuration with topic auto-creation

⚠️ Note: Kafka and Redis are currently wired in dependencies but not yet fully implemented in the application layer. This is a planned feature for upcoming releases.
